# iOS上架、硬件连通与候选人面试详谈

## 会议概览
- 主题：围绕即将于3-4月提交的 iOS 上架合规与审核准备、与硬件连接（蓝牙/Wi‑Fi）稳定性问题排查、订阅支付合规设计，以及候选人项目经验与匹配度的深聊。
- 参与：面试官（B）与候选人（A）。
- 背景：公司为美国主体的新创团队，面向欧美/日韩市场；当前样机已完成连通，存在重启后连接体验问题；计划上线订阅。

## 核心结论与重点
- iOS 上架高风险点集中在数据隐私合规、功能完整性、代码质量与性能/能耗控制。
- 审核需提供可用的测试账号与明确的隐私/加密说明；避免私有 API 与“功能过于简单”被拒。
- 蓝牙体验问题的关键在“进程重启后的状态判定 + 自动重连/恢复”，以及样机侧 KeepAlive/安全策略的协同。[idx-68]
- 订阅支付应遵循 IAP/Apple Pay 流程；避免第三方 H5 支付“结果回传缺失”的老问题，并打通测试扣费链路。
- AI 生成代码提升效率但容易埋下性能隐患（CPU/内存/能耗），审核会以能耗/性能为由拒审，需系统化 Profiling。[idx-32]
- 候选人具备多次全新 App 上架经验，但“一次性通过极少见”，通常 1 周内迭代可通过。

## iOS 上架要点（合规与审核）
- 隐私与数据安全
  - 对收集/存储的用户数据（如音频、账号属性等）进行明确说明；填写加密选项（出口合规相关），标注是否对敏感数据加密。[idx-13]
  - 海外偏好邮箱体系，尽量减少手机号依赖；数据分享需向用户明确披露风险与用途。[idx-20]
- 审核准备
  - 提供审核测试账号，保证“可用性完整体验”。
  - 功能完整度、交互与文案合规；避免功能过于简单被拒。
  - 代码质量：避免调用私有 API；遵循平台权限提示规范。
  - 性能/能耗：避免高 CPU/内存占用（审核可测出并据此拒绝）。
- 审核周期与通过率
  - 首次提交通常 5 个工作日内出结果；若超过一周多为设计层面问题。
  - 全新 App 多数会被打回 1-2 次；一周内修复、补齐功能/说明后再提可通过。

## AI 代码的风险与对策
- 常见隐患：监听/缓存设计不当导致高 CPU/内存；对场景考虑不全（只面向当前类）。
- 审核会以“过度耗电/性能差”拒审。
- 建议：
  - 系统化使用 Instruments/Profiler、Flutter/原生工具做性能与内存分析。
  - 对列表监听、缓存、定时任务等高频模块做专项优化与压测。
  - 代码评审纳入“私有 API/能耗/内存泄露/线程使用”检查清单。

## 订阅支付合规与踩坑
- 强烈建议遵循 Apple IAP/Apple Pay；海外常见信用卡/Apple Pay/Google Pay，第三方 SDK 成熟度高。[idx-60]
- 历史踩坑：第三方 H5 支付跳转后“结果回传缺失”，导致审核认为“功能不完善”并拒绝。
- 建议：
  - 明确订阅内购合规边界，避免绕过 IAP。
  - 准备沙盒/测试账号，打通扣费闭环（含 0.1 美元测试）。
  - 若接三方支付，确保回调/通知机制可靠与重试机制；日志全链路可追踪。

## 硬件连通（蓝牙/Wi‑Fi）问题分析与建议
- 现象：App 被 Kill 后重开需重新连接，竞品可直接恢复。
- 正确做法：
  - 进程启动后先通过系统蓝牙 API 查询设备连接状态（而非盲目重连），恢复 UI 与会话上下文；实现自动重连策略。[idx-68]
- 可能原因：
  - App 侧：未实现状态查询/重连逻辑，或蓝牙被关闭；蓝牙基类/状态机缺失。[idx-68]
  - 设备侧：KeepAlive/安全策略（如无心跳超时断开、限制重连窗口）导致断开；两端策略不协同。
- 建议落地：
  - 建立蓝牙“基类 + 状态机”，统一管理连接、重连、状态查询、异常恢复；前后台切换与进程重启专门分支。
  - 与固件对齐 KeepAlive 心跳频率、重连窗口、安全策略；提供心跳/断线事件上报。
  - 增加连接/断开/重连全链路日志，便于复现与快速定位。

## 数据传输与丢包处理经验
- 语音与指令：
  - 讯飞集成：多通过 Wi‑Fi 与云侧 API 交互，App 侧主要负责模式切换与触发。
  - 机器人/智能灯：音频多为“预下载-完整下发”，蓝牙/Mesh 传输控制/采样数据，不做实时流。[idx-88]
- 实时传输（边录边传）借鉴：
  - 协议：UDP/WebSocket；采用 CRC 完整性校验、丢包率阈值与插值弥合策略。[idx-96]
  - 业务策略：对轻微丢包容忍，超过阈值做补偿/重传；绘制/播放层容错平滑。

## 稳定性与监控
- 崩溃常见因：定时传输模块内存泄漏、构造不严谨导致崩溃。
- 监控手段：利用 App Store 崩溃信息 + 自建日志上传；过往项目在数百台规模下崩溃反馈约十条以内（未做严格统计）。[idx-102]
- 建议：上线前建立 Crash/ANR 基线指标与预警；关键链路增加守护与降级。

## 候选人经历与匹配度
- 经历概览：
  - 2010 年起从事 ERP 实施/开发（台资制造业，C/S 架构、.NET 技术栈等）；2017 年转移动端。[idx-111]
  - 全新 App 上架 10+ 次；一次性通过罕见，通常 1 周内迭代通过。
  - 支付集成（银行/第三方 SDK）、蓝牙/Mesh、机器人/智能灯等硬件相关项目经历。
  - 2023 年起广泛使用 AI（ChatGPT 等）辅助开发，并集成至 IDE。
  - 团队规模经验：8 人至 50+ 人；对初创持积极态度，认可公司面向 AI 前沿的产品方向。
- 其他：通勤松岗约 50+ 分钟；有家庭与子女；湖南籍等。

## 风险点与未决事项
- 蓝牙重连体验不达标：需尽快补齐“状态判定 + 自动重连 + 心跳/KeepAlive 协同”的设计与实现。
- 订阅/IAP 策略与测试闭环未实装：需明确内购合规方案，完成端到端测试（含审核运营账号）。
- 性能与能耗基线未建立：AI 代码引入后应进行专项性能检查与准入标准；上线前完成 Profiling 与压测。
- 隐私与加密选项需完善：App Store 提交前确保隐私说明、加密申报与数据处理合规。[idx-19]

## 建议的后续行动（与会议达成一致）
1. 在3月-4月初提交iOS App至App Store审核。
2. 准备并提供App Store审核测试账号。
3. 完善隐私与数据安全说明并配置数据加密选项。
4. 进行代码质量与性能检查，避免私有API与高CPU/内存耗用。
5. 优化蓝牙连接：实现进程重启后的状态判定与自动重连；评估样机KeepAlive策略。
6. 设计订阅支付遵循Apple内购/ApplePay，避免第三方网页回传缺失，并准备测试扣费流程。
