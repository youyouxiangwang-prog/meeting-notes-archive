# 把产品做到“能用”，再谈杀手功能与上线节奏

## 概要
团队围绕“先把产品做到能用，再追求杀手功能”的原则，统一了阶段性目标与时间节奏：优先补齐与打磨App基础能力，使真实用户在众筹前即可顺畅使用；并在此基础上用数据驱动验证潜在“杀手功能”（如Agent Mode），以批次Beta测试快速迭代，保障KOL样机与众筹如期推进。[idx-62]

## 关键讨论与共识
- 产品优先级
  - 由“差异化”转向“基础可用性”（稳定性、完整性、易用性），遵循二八原则：高频刚需优先于少数差异点。
  - Build 38当前可评约80分；去除无效入口、修复细节、提升稳定性与移动端呈现，可显著提升到90分以上。[idx-0]
- 基础问题举例（需尽快修复）
  - 仍保留的“AI task”占位应移除。
  - 分享总结中的符号/格式错误。
  - 长录音上传失败、文件丢失的稳定性问题。
  - 手机端呈现不友好、模板阅读体验不足等。
- UI/UX方向
  - 先“补全”缺页与统一风格，再考虑整体调性升级与品牌化差异。
  - 当前需补的模块页：模板引导页、设置页、检索、订阅后个人页等。
  - 去除临时“品牌色硬怼”的不一致风格，保证一致性与基础体验。
- 资源与节奏
  - 现有UI/UX人力偏兼职，迭代密度受限；需寻找长期合适人选，外包App开发作为阶段性补充。[idx-32]
  - 软件与硬件并行推进，但要在一张项目时间轴上对齐关键节点，避免软硬错位。[idx-434]

## 杀手功能与方法论
- 方法：从真实使用中挖掘，先提出假设→做出可用原型→投给真实用户→用行为数据检验（比调研更快更准）。[idx-71]
- 现有线索
  - “跨对话总结/写报告”已被用户验证有价值，且竞品（Pro等）未覆盖，是潜在差异点。[idx-63]
  - “与GPT/Slack打通”尚停留在概念演示，待落地验证。
- Agent Mode（车载场景）
  - 场景痛点：美国用户通勤车程长，车上希望“说了就办”以提升时间效率。
  - 拟定范围：语音识别意图→自动执行限定动作（写邮件、写报告、文字总结、给自己发邮件等），不对外发送，降低风险。[idx-125]
  - 技术判断：在App内可行；准确率不必100%，80–90%在车载高效场景可接受。
  - 交互建议：将语音输入整合进现有“Ask AI”的多轮对话入口，显著突出车载启用按钮；失败时需有回退与确认流程（收件人确认、保存/编辑/导出等）。
  - 硬件限制与取舍：BLE低功耗非长连接、CarPlay接入不确定；现阶段绕过硬件，优先以App方案实现（参考豆包的语音输入方式）。

## 测试与数据驱动
- 用户资源：广告积累的1300订阅、150–170定金，合计200–300可邀测试；100名活跃用户的数据即可快速验证。[idx-10]
- 数据埋点：按页面/点击/停留时长/转化等关键节点埋点，支持假设验证与迭代决策。
- Beta策略：分批邀请（先50，再扩大），持续一月，边开发边测试，快速闭环。

## 风险与阻碍
- 人力与效率：UI/UX兼职导致迭代密度不足；需补充长期稳定人选。
- 体验一致性：新增Agent功能可能引入交互复杂度，需谨慎设计失败回退与确认流程。
- 技术限制：BLE非长连接、CarPlay不确定；车载模式以App实现为主。
- 时间压力：边开发边测试，上架与众筹节点紧，需严格项目管理与缓冲。
- 物料依赖：说明书、包装与营销物料依赖UI定稿，需在项目表中明确联动与锁定点。
- 语音与意图识别：用户表达不清晰、识别错误率对体验影响大，需多轮对话与确认机制缓解。

## 决议与方案要点
- 聚焦“能用”的基础面：稳定录音、稳定总结、模板呈现舒适、分享无障碍、移动端友好。[idx-6]
- UI策略：先补全与统一，再做调性升级（品牌风格、层级感、引导页等后续迭代）。
- Agent Mode：以App内语音对话实现，多轮交互，限定执行范围，强调车载高效场景；原型尽快内测。
- Beta节奏：分批邀请，强埋点，数据驱动迭代；充分利用现有订阅与定金用户。
- 时间轴管理：软硬件与营销物料的关键点统一排期，确保KOL与众筹节点达成。
