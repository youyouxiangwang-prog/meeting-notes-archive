# SageMaker实战与机器学习工程化导论

## 概要
本次课程围绕“从业务问题到机器学习落地”的完整工程化流程展开，聚焦 Amazon SageMaker 的实操与企业级 MLOps 能力。上午以监督式学习为切入，系统讲解数据准备、模型训练、部署与监控的端到端闭环；同时结合工具链（SageMaker Notebook/Studio、内置算法、容器、ECR、Endpoints、Batch Transform、Data Wrangler、Glue、CloudWatch、Lambda 等）与工程实践（A/B 测试、蓝绿/金丝雀发布、弹性伸缩、成本优化、权限与自动化）进行演示与答疑。下午安排 Quick Labs 电信用户流失预测完整实操。

## 核心内容回顾

- 业务到问题定义
  - 以客服路由为例：业务目标是将用户来电准确路由至合适坐席 → 机器学习问题映射为多分类（监督式学习）。[idx-5]
  - 问题定义决定算法选择与数据需求，需数据科学家与业务专家共同梳理数据来源与标签。[idx-6]

- 数据准备与特征工程
  - 数据采集与整合：多系统（呼叫中心、用户中心、订单等）汇总至集中存储（S3/数据仓库），结构化数据可入数仓。[idx-7]
  - 数据清洗与对齐：样本、特征选择，缺失/异常处理，图像/文本等需格式统一。
  - 公开数据可用于预训练，自有数据做微调以提升业务适配度。

- 数据集划分与泛化
  - 训练/验证/测试集划分（如 80/10/10），用未见数据评估避免过拟合。[idx-17]
  - 关注模型泛化能力，训练完成后用测试集进行最终评估再部署。

- 训练、评估与部署
  - 多算法试验与调优，性能达标后评估→部署。
  - 部署形态：实时在线 Endpoint 或批量推理（Batch Transform）。[idx-136]
  - 生产级需求：压测与弹性扩缩（峰谷与大促流量场景），将模型视作软件组件纳入统一运维。[idx-28]

- 生产监控与闭环
  - 数据/概念漂移（data drift/concept drift）监控：对比生产统计与训练期统计，发现偏移触发重训。[idx-33]
  - 发布策略：A/B 测试、蓝绿/金丝雀发布，按流量配比比较新旧模型性能并逐步切换。

## SageMaker 实操与平台要点

- Notebook 实例
  - 一键启动、托管环境、多 Kernel（Python、TensorFlow、PyTorch 等），支持通过 SDK/CLI 远程触发训练与部署。[idx-144]
  - 建议开发期用 T3 系列（性价比高），需要再切换 GPU；注意关机与存储计费。[idx-171]

- 训练与推理（容器化）
  - 训练/推理均基于 Docker；自研算法容器推送至 ECR 后即可被训练/部署服务拉取执行。[idx-151]
  - 训练节点短生命周期（完成即释放），成本可控；支持 Spot 降本。[idx-175]
  - Endpoint 实时在线需手动删除才停止计费；支持自动扩缩与多模型负载均衡。

- 大数据与高吞吐
  - S3 File mode（默认）：先完整下载再训练；数据大时启动慢。[idx-146]
  - Pipe/Pipelined mode：边流式取 batch 边训练，适合千万级样本。[idx-96]
  - EFS/FSx：文件系统直挂载（NFS/高吞吐），相较 S3 拉取更快，适合高并发数据读取。[idx-99]
  - 批处理/分布式训练：按 batch 切分与多节点并行。

- 成本优化
  - 训练可用 Spot，大幅降本；推理成本往往占总成本大头（长期在线+请求量），Elastic Inference 以“算力片段”供给，降低 GPU 空置浪费。[idx-176]
  - 生命周期配置（Lifecycle config）：在“创建/启动”时自动执行脚本（拉取代码/数据、安装依赖），结合定时开关机/cron 进一步自动化。[idx-189]

- 权限与安全
  - IAM Role 赋权（更安全，临时凭证自动轮转）；可细粒度至 API 动作（Create/Start/Stop Notebook 等）。[idx-228]
  - 安全与便捷需权衡：多数据桶/新资源访问需走授权流程，建议标准化角色模板。

- Ground Truth 数据贴标
  - 支持图像/视频/文本（语音当前不支持）；结果存 S3（Manifest JSON）。
  - 自动贴标（Auto-Labeling）：小规模人工先标→训练弱监督模型→机器扩标→低置信度回流人工复核。[idx-119]
  - 众包无 SLA（超期自动结束），质量通过多工人投票与置信度衡量；对语音可考虑先用 AI 服务（如语音转文本）辅助生成弱标签。

- 边缘部署与模型优化
  - SageMaker Neo（基于 TVM）编译优化，降低边缘/终端资源占用。
  - 模型可转 ONNX，配合 Greengrass、DeepLens 等做本地推理与云边协同。

- 与开源/DevOps 的关系
  - 与 MLflow 等开源工具在理念上相通，SageMaker强调“托管+集成”，减少底层运维成本。
  - 机器学习 Pipeline 与软件 CI/CD Pipeline 可融合形成企业级 MLOps：数据/代码/模型三维变化的自动化治理。

## 现场答疑与团队痛点

- 现状与工具
  - 团队已在用 SageMaker Notebook；数据主要在 Redshift。
  - 困扰：Redshift→S3 抽取链路搭建复杂（权限/配置），已耗时一周。

- 建议方案（数据抽取与自动化）
  - 方案一：SageMaker Studio Data Wrangler 定义数据/特征抽取流程，结合 Feature Store 与定时任务（CloudWatch Events 触发）周期拉取落地 S3。[idx-76]
  - 方案二：Glue（Spark/Scala/Python）构建 ETL 任务，从数据源抽取转换后写入 S3；用 CloudWatch Events 触发 Lambda 调用 Glue 定时运行。[idx-85]
  - 会后提供 MLOps Workshop 示例链接（含 Glue ETL 示例，示例基于 S3 CSV，但可借鉴流程）。[idx-87]

- 大数据训练问题
  - 内存/报错：按 batch 训练、使用 pipelined mode，必要时采用分布式训练；数据在多处时考虑统一到高吞吐文件系统（EFS/FSx）。
  - 数据拉取与速度：S3 拉取慢于文件系统直挂载，示例与架构图将提供 GitHub 链接供实操参考。

- 其他要点
  - Notebook 生命周期脚本与权限实践、Git 集成、不能直接登录训练实例（托管模式）、日志与指标上报 CloudWatch。[idx-318]
  - 中国区与国际区功能差异：Ground Truth、Marketplace、边缘推理等在国内可见度/可用性不同。
  - Quick Labs 建议使用公司邮箱注册，避免自有账户资源/计费风险。
